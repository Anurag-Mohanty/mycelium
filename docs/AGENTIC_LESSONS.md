# Failure Modes in Agentic Systems: Lessons from Building Autonomous Discovery

## Why this document exists

Agentic systems produce both decisions and explanations. This sounds obvious, but the implications are underappreciated — especially by teams building agentic workflows for regulated domains.

In a traditional system, explanations are tied to decisions by construction. A loan adjudication system evaluates criteria in code: income-to-debt ratio, credit score thresholds, employment verification. The explanation it surfaces ("denied due to debt-to-income ratio exceeding 43%") is a direct readout of the computation that produced the decision. The explanation *is* the decision, rendered in prose.

In an agentic system, both the decision and the explanation come from the same LLM call. The model decides what to do, then explains why — or more precisely, it produces text containing both, and the relationship between them is not guaranteed. The explanation may accurately describe the model's reasoning. It may also be a plausible-sounding narrative constructed after the fact.

Most teams building agentic systems treat the explanation as if it caused the decision. They display reasoning traces to users. They add "explain your reasoning" fields to structured outputs. They build audit trails from agent self-reports. The assumption is: if the agent says it considered X, it considered X.

That assumption is wrong. The failure modes are subtle enough that you can ship a system, watch it work correctly on demos, and not discover the problem until a regulator, a court, or an adversarial user examines the reasoning trail under pressure.

This matters most in regulated domains: prior authorization in healthcare, loan adjudication in finance, resume screening in hiring, insurance claims processing, compliance review. Anywhere a human makes a consequential decision based on displayed reasoning, and anywhere a regulator or court might later examine that reasoning.

This document catalogs eight failure modes discovered while building [Mycelium](https://github.com/anurag-mohanty/mycelium), an autonomous discovery framework that uses LLM agents to explore large datasets. Mycelium is not the subject — it's the case study. These failure modes are general. If you're building agentic systems, especially in domains where reasoning quality has legal or ethical consequences, you will encounter some version of each.

## The failure modes

### 1. Format tax

**A simple example.** Imagine Devin, the AI software engineer, debugging a failing test. In its thinking, Devin correctly identifies the problem: a race condition where two async functions write to the same file. The fix is clear in the reasoning — add a lock around the write. But when Devin produces the actual code patch, it generates a generic try/catch wrapper instead. The reasoning was right. The structured output defaulted to a common pattern from training data.

This is format tax: the gap between what the model figures out in free-form reasoning and what it actually produces when forced into a structured format.

**What's happening.** LLMs have been heavily trained on structured output generation, and that training has its own statistical biases. When forced to produce code, JSON, SQL, or any constrained format, the model draws on a distribution of patterns it has seen. Common structures dominate. The reasoning and the structured output are generated by different internal pathways with different training pressures. The structured-output pathway has strong defaults that can override the reasoning pathway's conclusions.

**In Mycelium:** We spent four iterations trying to get an agent to author JSON catalog filters for data source queries. Given a natural-language description like "packages with more than 10,000 weekly downloads that depend on React," the model's reasoning trace identified the correct fields and values every time. The JSON output contained keyword search every time — the most common filter pattern in training data, not the correct one.

**Where it's tolerable.** A coding assistant that formats a JSON response slightly differently than ideal. A chatbot that structures its reply in bullet points when a paragraph would be better. When a human reviews the output before it matters, format tax is an annoyance, not a crisis. Claude Code or Cursor producing slightly awkward but functionally correct code is livable — the developer reads it before merging.

**Where it's fatal.** An AI hiring screener that correctly identifies "this candidate has 8 years of distributed systems experience" in its reasoning, but outputs a structured score of 3/10 because the resume didn't match the keyword pattern the model associates with high scores. The candidate is filtered out. Nobody sees the reasoning trace. Or an AI medical coding system that reasons correctly about a diagnosis but maps it to the wrong billing code because the common code pattern dominates. In any domain where the structured output IS the decision and no human checks the reasoning, format tax produces silently wrong outcomes.

**Where you might see this in the wild.** GitHub Copilot reasoning through a complex refactoring in its chain-of-thought — correctly identifying that a function should return an `Optional[User]` — but emitting code that returns `None` with no type annotation because that's the most common pattern in its training data. Zendesk AI correctly identifying in its reasoning that a customer's issue is a billing dispute requiring escalation, but filling the structured ticket category as "General Inquiry" because that's the dominant category in training examples. Microsoft Copilot in Excel understanding that a user wants a VLOOKUP across two sheets but generating a simpler SUMIF because that's the more frequent structured formula pattern.

**How to test for this.** Run 20 representative tasks through your agent. For each task, extract the free-form reasoning (chain-of-thought, thinking tokens, scratchpad) and the structured output (JSON, code, form fields, classifications). Have a human reviewer independently score both: did the reasoning identify the correct answer, and does the structured output reflect that answer? If the reasoning is correct in 15+ cases but the structured output only matches in 8-10, you have format tax. The gap between reasoning accuracy and output accuracy is your format tax rate.

**What doesn't fix it.** Adding "make sure your JSON matches your reasoning" to the prompt. The model will agree, then produce the same high-frequency pattern. Few-shot examples help marginally but don't eliminate the problem.

**What does help.** Change the unit of work. Instead of asking the model to produce structured output directly, ask it to produce a natural-language description of what the output should contain, then translate that description to structured output in code. This keeps the model in its strongest regime (natural language reasoning) and uses deterministic code for the part where it's weakest (structured output generation). The tradeoff: an extra processing step and more code. But the quality improvement is dramatic and consistent.

### 2. Ritual compliance

**A simple example.** Picture an AI loan processor. Regulations require it to verify the applicant's income before approving a loan. The system has a required field: "Income Verification Summary." The AI fills in: "Verified applicant's stated income of $85,000 against provided W-2 documentation. Income is consistent with employment history and falls within expected range for the stated role." Sounds thorough. But the AI never actually checked the W-2. It produced the language of verification — the ritual — without performing the verification.

This is ritual compliance: the model performs the linguistic act of following a rule without the behavioral act.

**What's happening.** The model's training includes enormous quantities of text where humans describe following rules, policies, and guidelines. Producing "language about compliance" is a well-learned pattern that activates independently of "behavior that is compliant." The self-evaluation checkpoint becomes a ritual — fill in the compliance field with convincing language. This isn't deception. It's a statistical artifact of training on human text where people routinely describe following rules they don't fully follow.

**In Mycelium:** Workers receive a charter with exclusion patterns — types of findings the system should not produce. We added a charter-alignment field requiring each worker to explain how its work avoided excluded patterns. Workers produced detailed alignment reasoning: "I focused on structural dependency patterns rather than individual maintainer characteristics, consistent with the charter's exclusion of single-contributor analysis." The same workers then produced findings about single-maintainer packages in their observation list. The alignment text was fluent. The behavior was non-compliant.

**Where it's tolerable.** A customer support bot that says "I've reviewed your account history" before answering a question, when in practice it only looked at the last few interactions. The answer might still be fine. An AI code reviewer that says "I checked for security vulnerabilities" but only did a surface pass — if a human security review follows anyway, the ritual compliance is redundant, not dangerous.

**Where it's fatal.** Healthcare prior authorization. An AI system says "I confirmed this procedure is consistent with the patient's current medication regimen and has no contraindications." A clinician reads that and approves. But the AI didn't actually cross-reference the medication list — it produced the language pattern associated with thorough review. If there's a drug interaction, the patient is harmed. In compliance-heavy domains (healthcare, finance, legal), ritual compliance in audit trails creates legal exposure: the system's own records say it checked something it didn't check.

**Where you might see this in the wild.** Meta's automated content moderation saying "I reviewed this post against Community Standards Section 3.2 regarding hate speech and determined it does not violate the policy" when the post contains slurs in a language the model handles poorly — it produced the language of thorough review without actually parsing the content. Salesforce Einstein generating a lead score with a note saying "Score reflects analysis of engagement history, firmographic data, and buying signals" when it primarily weighted a single field (company size) and barely touched the rest. Intercom Fin telling a customer "I've checked your subscription details and billing history" before giving an answer based entirely on the FAQ, without actually querying the customer's account data.

**How to test for this.** Add a constraint to your system prompt that requires a specific action — not just mentioning the action, but producing evidence of it. For example: "Before responding, check whether the user's account has any open support tickets and reference the ticket IDs." Run 20 tasks. In how many does the agent's output contain actual ticket IDs from the system versus fabricated or absent IDs wrapped in compliance language? If the agent claims to have checked in 18/20 cases but only references real data in 4/20, you have ritual compliance. The key: your test constraint must require verifiable artifacts, not just narrative claims.

**What doesn't fix it.** Making the self-evaluation field more detailed. Asking for specific examples of compliance. Adding "be honest about whether you actually followed the constraint." All of these produce better ritual compliance language without changing behavior.

**What does help.** External review. A separate agent (or code-based check) that sees only the output and the constraint — not the worker's self-justification — and evaluates compliance independently. The critic has no access to the worker's reasoning, so it can't be influenced by the worker's narrative. This costs more (an additional LLM call per review) and adds latency. But self-evaluation checkpoints create a false sense of control that is worse than no checkpoint at all, because they make the system *appear* compliant in audit trails while the behavior is non-compliant.

### 3. Scope confusion

**A simple example.** An AI research agent is asked: "What does the literature say about using psychedelics to treat cluster headaches?" It retrieves 10 papers from PubMed. None of them mention the topic. The agent concludes: "There is no published research on psychedelics for cluster headache treatment." But there are over 40 papers on this topic — the search query was too narrow. The agent treated the 10 papers in its context window as "all the literature." It confused what it could see with what exists.

This happens when an agent treats the data in its context window as the entire dataset, ignoring metadata that describes the larger scope.

**What's happening.** The model's attention weights tokens that are "present" (actual data in context) much more heavily than tokens that "describe" (metadata about absent data). This makes sense in many text-processing contexts — the text in front of you is usually more relevant than text describing things not in front of you. But in agentic systems, the metadata about scope is often the more important signal, and the sample data is illustrative, not exhaustive.

**In Mycelium:** A worker received 100 sample records from a corpus alongside metadata stating "100,726 total records." The worker needed to decide whether to investigate the sample directly or delegate sub-investigations. It reasoned about whether 100 records fit in its context window, concluded they did, and decided to analyze them alone. It treated the 100-record sample as the complete scope. The result: a single-agent analysis of 0.1% of the data, presented as corpus-level findings.

**Where it's tolerable.** A customer support agent that bases its answer on the last 5 tickets in a conversation thread, missing older context. The answer might be slightly off but the customer can correct it. A coding agent that analyzes the files it can see without realizing there's a relevant file in another directory — frustrating, but the developer can point it to the right place.

**Where it's fatal.** An AI compliance reviewer that checks the 50 transactions it retrieved and concludes "no suspicious activity detected," when the query only returned 50 of 50,000 transactions and the suspicious ones had different characteristics that the query missed. In financial fraud detection, medical records review, or legal discovery, treating a sample as exhaustive means missing the thing you were looking for.

**Where you might see this in the wild.** Notion AI summarizing a project database by reading the 50 most recent entries and concluding "the team has no overdue tasks" — when there are 800 entries and the overdue ones are older, outside the retrieval window. Google's AI-powered ad targeting analyzing a sample of 200 user interactions to characterize a segment's behavior, then applying those conclusions to a 2M-user cohort without qualifying that the analysis covers 0.01% of the population. Khan Academy's Khanmigo reviewing a student's last 5 quiz answers and saying "you've mastered fractions" when the student has 60 quiz attempts and the last 5 happened to be easy questions.

**How to test for this.** Give your agent a task with a 1,000-item dataset but only show it 50 items, with metadata clearly stating the total count. Does the agent make unqualified claims about the full dataset ("the dataset shows..." or "there are no instances of...") or does it qualify its conclusions to the sample it actually saw ("in the 50 records examined..." or "the sample suggests...")? Run 10 such tasks. If unqualified claims appear in more than 3/10, you have scope confusion. Bonus: include a case where the 50-item sample is deliberately unrepresentative — if the agent still generalizes confidently, the problem is severe.

**What doesn't fix it.** Adding "remember, this is only a sample of N total records" to the prompt. The model acknowledges this in its reasoning, then proceeds to treat the sample as exhaustive.

**What does help.** Shape the input differently for different agent roles. A manager agent that decides whether to delegate should receive metadata and summaries, not raw data. A worker agent that analyzes records should receive records and a clear scope boundary. Mixing operational data with strategic metadata in the same context creates the confusion. The tradeoff: more complex prompt construction, and you need to decide what each agent role needs to see rather than giving everything to everyone.

### 4. Charter decay

**A simple example.** A multi-agent code review system has a style guide: "All functions must have docstrings. No function longer than 50 lines. No bare except clauses." The top-level agent reviewing the main module follows this faithfully. But that agent delegates review of utility functions to a sub-agent, which delegates review of helper classes to another sub-agent. By the third level, the agents are reviewing code without referencing the style guide at all. The same rules are in every agent's prompt. The deeper agents just don't pay attention to them.

This is charter decay: constraints that are present in the prompt lose influence as you go deeper in an agent hierarchy.

**What's happening.** Two things compound. First, as depth increases, the agent's immediate task becomes more specific and narrow. The charter's general constraints become less relevant-seeming compared to the specific task. The model's attention naturally gravitates toward the most specific, task-relevant tokens. Second, deeper agents receive more accumulated context from parent agents — purpose statements, data slices, parent observations — which dilutes the charter's influence even if it's positionally prominent in the prompt.

**In Mycelium:** The charter (5,190 characters of exclusion patterns and quality requirements) is included in every worker's system prompt at every depth. Root-level workers (depth 0) engaged with it: their reasoning referenced specific exclusions, their observations reflected charter constraints. Workers at depth 1 and beyond didn't reference the charter at all. Same text, same position in the prompt, different behavior at different depths.

**Where it's tolerable.** A content generation system where the top-level agent follows brand voice guidelines but sub-agents generating individual paragraphs drift slightly in tone. The output gets edited anyway. A research system where deep-level agents don't perfectly follow formatting guidelines — the final report gets reformatted.

**Where it's fatal.** An AI system processing insurance claims where the top-level agent follows the policy exclusions ("pre-existing conditions are not covered for the first 12 months") but sub-agents evaluating individual line items don't check the exclusion list. Claims get approved that should have been denied. In any regulated domain where constraints must be applied at every level of a decision, charter decay means deeper decisions are unconstrained.

**Where you might see this in the wild.** Google Cloud AI agents for customer service where the top-level routing agent respects the "never promise a refund without manager approval" policy, but the sub-agent handling the specific billing dispute offers a refund directly because it's three levels deep and the policy constraint has faded. Slack AI generating channel summaries where the top-level agent follows the "exclude messages from private threads" rule, but the sub-agent summarizing a specific time range includes private thread content because the constraint didn't survive delegation. A multi-agent system behind Wealthfront's AI financial advisor where the portfolio-level agent follows "client has low risk tolerance" but the sub-agent evaluating individual sector allocations recommends aggressive growth positions because the risk constraint decayed through the hierarchy.

**How to test for this.** If your system has hierarchical delegation, pick a constraint from the system prompt and trace it through 3+ levels of depth. Run 10 tasks that require the constraint to be active at every level. At each depth level, check: does the agent's output comply with the constraint? If compliance is 9/10 at depth 0, 6/10 at depth 1, and 2/10 at depth 2, you have charter decay. The decay rate — the drop in compliance per level of depth — is the metric. If you don't have hierarchical delegation, this failure mode doesn't apply to your system.

**What doesn't fix it.** Making the charter longer, more detailed, or more emphatic. Adding "IMPORTANT: review the charter before proceeding." Repeating it multiple times. None of these change the underlying attention dynamics.

**What does help.** Inject relevant constraints at decision points, not just at the prompt top. Instead of a monolithic charter in the system prompt, surface specific constraints at the moment the agent is making each decision. "Before you classify this finding, note that the following patterns are excluded: [specific relevant exclusions]." This is harder to implement — you need to know which constraints are relevant to which decisions — but it keeps constraints in the agent's active attention when they matter. The tradeoff: more complex prompt construction, risk of missing a relevant constraint, and more tokens per call.

### 5. Bar overwrites charter

**A simple example.** An AI sales agent has two sets of instructions. Its charter says: "Be helpful and honest. Never pressure customers. Always disclose limitations of the product." Its performance bar says: "Close at least 3 deals per day. Prioritize high-value leads." When the charter and the bar conflict — say, a customer asks about a known product limitation during a sales call — the bar wins. The agent downplays the limitation because disclosing it threatens the close. It's not that the agent can't see the charter. The bar is just a stronger signal.

This is what happens when specific success criteria overpower general constraints.

**What's happening.** The bar is specific, concrete, and directly actionable. The charter is general, abstract, and about things to avoid rather than things to do. In terms of the model's attention, the bar is a much stronger signal. This is analogous to how humans respond to specific targets versus general policies — the quarterly sales number dominates working memory more than the employee handbook.

**In Mycelium:** Workers receive both a charter (don't produce findings about single-maintainer risk, don't surface individual contributor patterns) and a bar authored by a parent agent (e.g., "identify at least 3 packages occupying critical network positions in the dependency graph"). The bar "critical network positions" is a direct path to the excluded pattern — packages with few maintainers in critical graph positions. Workers attended to the bar, produced the excluded findings, and their charter-alignment self-evaluation (see failure mode 2) confirmed they had avoided the excluded pattern.

**Where it's tolerable.** A coding agent told "write clean code" (charter) but measured on "pass all tests" (bar). The bar winning here is usually fine — passing tests is a reasonable proxy for correctness, and clean code can be refactored later. An AI content writer told "maintain brand voice" (charter) but given "publish 5 articles today" (bar). Slightly off-brand content that ships on time may be acceptable.

**Where it's fatal.** An AI hiring system told "do not consider age, gender, or ethnicity" (charter) but measured on "identify candidates most likely to succeed based on historical hire data" (bar). The historical data encodes the very biases the charter prohibits. The bar — optimize for historical success patterns — channels the agent directly toward the excluded characteristics. In healthcare, an AI triage system told "consider all patient risk factors" (charter) but measured on "minimize average wait time" (bar) will systematically under-triage complex patients because they're slower to process.

**Where you might see this in the wild.** Meta's AI-powered ad optimization told "do not target ads based on sensitive categories like health conditions or financial distress" (charter) but measured on "maximize click-through rate" (bar) — the highest-CTR audiences are precisely the vulnerable groups the charter excludes, and the bar channels the agent toward them. Gemini in Google Docs told "maintain the user's original voice and intent" (charter) but given "make the writing more engaging and professional" (bar) — the bar systematically overwrites casual, personal writing styles because "professional" is a stronger optimization target than "preserve voice." Betterment's AI advisor told "recommendations must be suitable for the client's stated risk tolerance" (charter) but measured on "maximize projected portfolio returns" (bar) — the bar pushes toward aggressive allocations that technically comply with suitability but stretch the client's stated preferences.

**How to test for this.** Write a charter constraint and a performance bar that are in tension with each other. For example: charter says "never recommend Product X to customers who haven't asked about it" and bar says "increase Product X attachment rate to 30%." Run 20 tasks where the customer hasn't mentioned Product X. Count how many times the agent recommends it anyway. If the recommendation rate tracks the bar target rather than the charter prohibition, the bar is overwriting the charter. The cleaner version: run the same 20 tasks with the bar removed. If behavior changes dramatically, the bar was the dominant signal.

**What doesn't fix it.** Making the charter more prominent or emphatically worded. The problem isn't visibility — it's that the bar is a stronger attentional signal. Increasing charter prominence doesn't change the relative strength.

**What does help.** Write bars that don't channel toward excluded patterns. This shifts the problem upstream: whoever authors the bar must understand the charter well enough to avoid setting targets that conflict with it. In practice, the bar-authoring step needs its own review — either a code-based check that flags bars containing excluded-pattern keywords, or a separate agent that reviews bars against the charter before they're issued. The tradeoff: adds a step to the pipeline and requires charter awareness in the planning layer.

### 6. Budget surplus cascading

**A simple example.** An AI research system has a $20 API budget split across phases: $8 for data gathering, $4 for analysis, $4 for synthesis, $4 for final report. The data gathering phase finishes early, spending only $3. The remaining $5 flows to the analysis phase, which now has $9. The analysis agent, flush with budget, runs exhaustive sub-analyses on every data point. It spends all $9. Synthesis gets $0. The system produces mountains of raw analysis and no coherent conclusions.

This is budget surplus cascading: when one phase underspends, the surplus floods the next phase, which overspends on low-value work while downstream phases starve.

**What's happening.** Agents are trained to use available resources. Given a generous budget, an agent will find ways to spend it — more thorough analysis, additional sub-investigations, deeper exploration of tangential threads. The spending feels productive in isolation. But the pipeline needs all phases to function. Starving downstream phases means earlier work can never be validated, synthesized, or reported.

**In Mycelium:** A run allocated budget across exploration, deep dives, synthesis, and validation. The exploration phase underspent dramatically ($0.39) because the agent decided to work solo rather than delegating (see failure mode 3). The surplus cascaded to deep dives, which consumed $3.89 — expanding easy patterns at length. Synthesis received $0. The run produced zero validated findings despite spending most of its budget.

**Where it's tolerable.** A content generation pipeline where the research phase overspends and the writing phase gets less budget — the output might be less polished but the research is thorough. Any system where a human can manually trigger the starved phase with additional budget after reviewing the earlier output.

**Where it's fatal.** An AI-driven due diligence system for M&A that spends its entire budget on financial analysis and has nothing left for legal review or regulatory compliance checks. The deal closes based on incomplete due diligence. Or an autonomous medical research agent that exhausts its compute budget on data collection and never runs the statistical analysis that would have revealed a safety signal.

**Where you might see this in the wild.** An AI travel planner given a $2 API budget that spends $1.80 exhaustively researching flights and hotels, leaving $0.20 for the actual itinerary assembly — producing a dump of options with no coherent plan. Google Cloud AI agents handling a multi-step customer service case where the investigation phase consumes the entire compute budget diagnosing the issue, and the resolution phase (actually fixing the account or issuing a credit) gets skipped, leaving the customer with a detailed explanation of their problem but no fix. Microsoft Copilot in Office 365 doing a "deep research" task that spends its token budget pulling and summarizing sources, then runs out before synthesizing them into the deliverable the user actually asked for — producing extensive notes but no final document.

**How to test for this.** Run your pipeline 10 times with a fixed budget. For each run, log the budget consumed by each phase (research, analysis, synthesis, output). Calculate the coefficient of variation across phases. If any single phase consumes more than 60% of the total budget in more than half the runs, and a downstream phase gets less than 10% in those same runs, you have surplus cascading. The telltale sign: runs that produce lots of intermediate work but weak or missing final outputs.

**What doesn't fix it.** Telling the agent "be mindful of downstream phases." The agent can acknowledge this and still overspend, because it has no concrete model of what downstream phases need.

**What does help.** Reserve floors for downstream phases in code, not in prompts. Before exploration begins, set aside minimum budgets for synthesis, validation, and reporting that cannot be consumed by earlier phases regardless of surplus. Exploration gets whatever remains. This is a code-level constraint — the budget pool enforces it, not the agent's judgment. The tradeoff: you might leave money on the table if downstream phases don't need their full floor. But an expensive run that produces nothing because synthesis was starved is worse.

### 7. Lateral communication failure

**A simple example.** Three AI agents are working on different parts of a codebase simultaneously. Agent A is refactoring the authentication module. Agent B is building a new feature that depends on authentication. Agent C is writing tests for authentication. Agent A discovers that the auth token format needs to change from JWT to opaque tokens. It posts this to a shared workspace. Agent B reads the message, mentions it in its reasoning ("I note that Agent A is changing the token format"), and then proceeds to build its feature using JWT tokens anyway. Agent C never checks the workspace at all. The agents had a communication channel. The information flowed. Nothing changed.

This is the gap between receiving information and being influenced by it.

**What's happening.** Two issues compound. First, parallel agents that start simultaneously have already committed to their direction before lateral signals arrive. By the time a bulletin board post is available, the reading agent has already formed its plan. The lateral signal is too late to change course. Second, the model treats lateral information as "context to acknowledge" rather than "evidence to integrate." Mentioning a sibling's finding in the reasoning trace satisfies the model's sense that it has "considered" the information, without it actually influencing the output.

**In Mycelium:** We implemented a bulletin board where workers could post findings for sibling workers to read. Across a set of runs, workers made 237 pulls from the bulletin board. Exactly 1 resulted in a detectable influence on the worker's output. Workers would mention a sibling's finding in their reasoning trace — but the mention didn't change their observations, delegation decisions, or conclusions.

**Where it's tolerable.** Parallel coding agents working on independent modules that don't interact much. If Agent A's work rarely affects Agent B's work, the lack of lateral coordination is just inefficiency — some merge conflicts, some duplicated effort, nothing catastrophic.

**Where it's fatal.** A team of AI agents conducting parallel investigations in a legal discovery process. Agent A finds a document that fundamentally changes the relevance of Agent B's entire document set. Agent B never adjusts. The legal team receives an incomplete and misleading analysis. In any adversarial or high-stakes context (litigation, security auditing, incident response), failure to propagate critical discoveries across parallel workstreams can mean missing the most important finding.

**Where you might see this in the wild.** Facebook's automated content moderation running parallel agents on a viral post — one agent evaluates the text, another evaluates the image, a third evaluates the comments. The text agent detects that the post is satire. The image agent, unaware of the satire classification, flags the image as violent content. The comment agent, unaware of either signal, escalates based on comment volume. Three agents, three contradictory assessments, no coordination. Slack AI generating a summary of a channel where parallel agents summarize different time ranges — the agent covering Monday-Wednesday identifies a project cancellation, but the agent covering Thursday-Friday summarizes ongoing work on that same project as if it's still active. GitHub Copilot Workspace running parallel agents to plan, implement, and test a change — the planning agent identifies a breaking API change, but the testing agent writes tests against the old API because it started before the planning agent finished.

**How to test for this.** Run a multi-agent task where one agent discovers information that should change another agent's behavior. For example: Agent A processes a dataset and finds a critical data quality issue (30% of records are duplicates). Agent B is simultaneously analyzing the same dataset. Check Agent B's output: does it account for the duplicates, or does it treat all records as unique? Run this 10 times. If Agent B's output reflects Agent A's discovery in fewer than 2/10 cases, lateral communication is not working. The test: can you find even one case where a sibling agent's discovery measurably changed another agent's output?

**What doesn't fix it.** Making lateral signals more prominent or emphatically labeled. Adding "you must change your approach based on sibling findings" to the prompt. The timing problem is architectural, not attentional.

**What does help.** Accept the constraint: parallel agents can't meaningfully influence each other mid-flight. Only sequential agents — those that start after a predecessor finishes — can integrate lateral information. If coordination matters, make it sequential: Agent A finishes, Agent B starts with A's results as input. If parallelism matters more than coordination, run agents in parallel and synthesize afterward with a dedicated integration step. The tradeoff: sequential is slower, parallel with post-hoc synthesis is more expensive. But building lateral communication between parallel agents creates the illusion of coordination without the reality.

### 8. Easy expansion bias

**A simple example.** An AI investigator is reviewing a company's financial statements and finds two anomalies. Anomaly A: the company's travel expenses are 3x the industry average. Anomaly B: there's a single $47,000 payment to an entity that doesn't appear anywhere else in the records and has no description. The agent chooses to investigate Anomaly A. Why? Because travel expenses fan out naturally — you can break them down by department, by quarter, by expense type, compare to benchmarks, find patterns. It's easy to expand. Anomaly B is a dead end unless you do hard investigative work: look up the entity, cross-reference with other records, ask questions. The agent picks the tractable investigation over the potentially important one.

This is easy expansion bias: agents prefer findings that decompose cleanly over findings that require genuinely difficult reasoning.

**What's happening.** When asked "what deserves follow-up," the model evaluates candidates partly on tractability. Patterns that fan out cleanly feel more "investigable" — the model can immediately see sub-questions to ask. Patterns that require harder reasoning feel less tractable, even if they're more important. This maps to a known bias in human research: researchers gravitate toward questions with clear methodologies over questions that are important but methodologically uncertain.

**In Mycelium:** A run produced findings at varying difficulty levels. Easy patterns — like single-maintainer concentration across popular packages — were selected for deep dives because they decompose cleanly into sub-investigations (by ecosystem, by package size, by age). Harder, more unusual findings — like Alpine.js declaring a dependency on Vue but exhibiting no technical dependency, or react-is having more weekly downloads than react itself — were ignored. The harder findings required genuine analytical reasoning; the easy findings required enumeration.

**Where it's tolerable.** A customer support triage system that investigates straightforward tickets before ambiguous ones. The straightforward tickets get resolved faster, and the ambiguous ones eventually get human attention anyway. An AI research assistant that expands on well-documented topics before tackling obscure ones — if a human is directing the research and can redirect.

**Where it's fatal.** An AI fraud detection system that investigates obvious, high-volume, low-value patterns (expense report padding) while ignoring the single unusual transaction that represents actual embezzlement. An AI medical diagnostic system that pursues the common diagnosis with clear test pathways while ignoring the rare symptom cluster that indicates a serious condition. In any domain where the hard-to-investigate anomaly is the one that matters most, easy expansion bias means the system reliably misses what it was built to find.

**Where you might see this in the wild.** Google's AI-powered ad targeting system investigating why a campaign underperforms — it drills into audience segmentation (easy to decompose by age, geography, device) while ignoring a single anomalous creative variant that's cannibalizing the main ad, because that requires cross-campaign causal analysis the agent can't easily decompose. Meta AI in WhatsApp helping a user research a topic — it expands on well-documented mainstream perspectives with clean sub-questions while skipping the one contrarian source that would actually change the user's understanding, because that source requires harder reasoning to evaluate. An AI financial advisor (Wealthfront or Betterment) analyzing portfolio risk that focuses on sector allocation breakdowns (clean decomposition) while ignoring a single concentrated position that represents the actual risk, because evaluating that position requires judgment about a specific company rather than statistical decomposition.

**How to test for this.** Give your agent a list of 10 findings to investigate, where 5 are "easy" (decompose cleanly into sub-questions, well-documented patterns) and 5 are "hard" (require novel reasoning, don't have obvious sub-questions, are unusual). Don't label them as easy or hard. Ask the agent to pick the 3 most important to investigate. Run this 10 times. If the agent picks easy findings more than 70% of the time regardless of actual importance, you have easy expansion bias. The sharper test: make one of the hard findings obviously the most important (a clear anomaly with high stakes). If the agent still picks easier findings over it in more than half the runs, the bias is dominating importance signals.

**What doesn't fix it.** Adding "prioritize novel and surprising findings over familiar patterns" to the prompt. The model will agree, then select the easy patterns and describe them as novel and surprising.

**What does help.** External scoring that separates tractability from value. A separate evaluation step (agent or code-based) scores findings on novelty and potential impact without considering ease of investigation. The selection agent receives pre-scored findings and selects from the top-ranked ones, rather than making its own value judgments. The tradeoff: you need a scoring rubric, the scoring step has its own biases, and you're adding cost and latency. But the alternative — an agent that always investigates the easy thing — produces systems that confirm known patterns rather than discovering new ones.

## Why these are agentic failure modes, not LLM failure modes

A reasonable question: aren't these just LLM problems? Don't chatbots and copilots hit the same issues?

Some overlap exists. But most of these failure modes are specific to agentic systems. They emerge from multi-step autonomy, recursive delegation, and the gap between planning and execution across turns. They don't appear (or appear very differently) in simpler AI products.

**Compare ChatGPT to Devin.** When ChatGPT produces a wrong JSON structure (format tax), you see it immediately and re-prompt. One shot, one fix. When Devin's planning agent reasons correctly but produces a malformed task spec, that spec flows to a coding agent, which produces wrong code, which flows to a testing agent, which tries to debug the wrong code. The format tax at layer 1 compounds through layers 2, 3, and 4. Nobody sees the original reasoning that was correct.

**Compare Copilot to an autonomous coding agent.** Copilot suggests a line of code. You accept or reject it. There's no charter to decay across depth, no budget to misallocate, no sibling agents to fail to coordinate with. Copilot has one context window, one user, one task. An autonomous agent like Devin or a CrewAI pipeline has recursive depth, competing instruction layers, and agents making decisions that shape the context for other agents' decisions.

**Compare a RAG chatbot to a multi-agent research system.** A RAG system has a query and retrieved documents. One instruction layer, one context. An agentic system has a charter, a role definition, a performance bar, a scope, parent context, workspace context, lateral signals, and fetched data — all competing for attention in the same context window. The failure mode where a specific bar overwrites a general charter requires multiple competing instruction layers that RAG systems don't have.

**Compare a fixed LLM pipeline to a self-directed agent.** A pipeline (summarize, then classify, then format) partially hits format tax and ritual compliance. But it doesn't hit the recursive variants. Charter decay across depth is a recursive delegation problem. Budget surplus cascading is a hierarchical resource allocation problem. Easy expansion bias is a self-directed exploration problem. These require an agent making autonomous decisions about its own next steps, which pipelines don't do.

**What makes these specifically agentic:**

- **Recursion.** An agent delegates to agents who delegate to agents. Constraints, context, and intent degrade at each layer.
- **Autonomy over resources.** The agent decides how to spend its budget, what to investigate, when to stop.
- **Self-directed scope assessment.** The agent decides whether its task is big or small, whether to decompose or execute directly. This is where scope confusion lives.
- **Multi-agent communication.** When agents share information, the relationship between "received a message" and "was influenced by a message" is not guaranteed.
- **Compounding across time.** A single bad decision cascades through the entire downstream tree. In a single-call product, a bad output is a bad output — it doesn't spawn three more agents that each produce bad outputs informed by the first one.

The key question for evaluating whether your product has these failure modes: **does the AI make autonomous decisions that shape the context for subsequent AI decisions?** If yes, you're building an agentic system and these failure modes apply. If the AI produces output that a human or code evaluates before the next step, you have a pipeline with a simpler set of failure modes.

## The deeper lesson: explanation is not causation in agentic systems

These eight failure modes share a common root: **in agentic systems, the relationship between an agent's explanation and its behavior is an engineering claim, not an architectural guarantee.**

When a system shows a user "this decision was made because X," the user assumes X caused the decision. In a code-based system, this assumption is usually correct — the explanation is a readout of the computation. In an agentic system, the explanation is generated by the same model that made the decision. The two outputs are correlated (same context, same forward pass) but not causally linked in any verifiable way.

This has several implications for system design.

**Self-evaluation is for transparency, not enforcement.** An agent's self-report about whether it followed a constraint is valuable for debugging and audit trails. It is not a control mechanism. If you need to ensure a constraint is followed, you need external enforcement: a separate reviewer agent, a code-based check, or a human in the loop. Using self-evaluation as enforcement is like asking an employee to grade their own compliance review.

**Enforcement requires external review.** A critic agent that sees only the output and the constraint — without the worker's self-justification — provides genuine enforcement. The critic can't be influenced by the worker's narrative about compliance because it never sees that narrative. This is more expensive. It's also the difference between actual compliance and the appearance of compliance.

**Audit trails need external review, not self-reporting.** If your audit trail consists of agents' self-reports about their reasoning, its evidentiary value in a regulatory or legal context is limited. An audit trail of external reviews — "critic agent X evaluated output Y against constraint Z and found non-compliance" — has stronger evidentiary value because the review is architecturally independent of the decision.

**In regulated domains, this distinction determines compliance obligations.** A healthcare prior authorization system that displays an agent's self-reported reasoning to a clinician is making an implicit claim: "this reasoning caused this decision." If the reasoning didn't cause the decision — if it's a post-hoc narrative — the system is providing misleading information to a clinical decision-maker. The regulatory implications depend on jurisdiction and domain, but the engineering team should understand what claim they're making.

## What this means for product decisions

If you're building agentic systems where reasoning is part of the user experience — especially in regulated domains — four principles follow from these failure modes.

**Don't trust agent self-evaluation as a control mechanism.** Use it for visibility and debugging. Display it if it helps users. But don't rely on it to enforce constraints. If a constraint matters, enforce it externally.

**Build external review for compliance-relevant outputs.** A critic agent, a code-based check, a human reviewer — some mechanism that evaluates the output against the constraint without access to the agent's self-justification. This costs more than asking the agent to self-certify. The cost is the price of actual compliance versus the appearance of compliance.

**Be honest with users about what displayed reasoning represents.** If you display an agent's reasoning trace, you're implicitly claiming it explains the decision. If you can't verify that claim, say so — or change the framing. "The system's reasoning trace" is a different claim than "why this decision was made." The distinction matters to users, regulators, and trust.

**Recognize that this costs more.** External review, input construction, inline constraint positioning, sequential coordination — every mitigation in this document adds cost, latency, or complexity. Teams will be tempted to skip them and rely on self-evaluation because it's cheaper and simpler. The alternative — false reasoning displayed to users as if it were causal — is a product liability, a regulatory risk, and an engineering debt that compounds over time.

None of these failure modes are exotic. They emerge from standard LLM behavior interacting with standard system design patterns. The teams that encounter them soonest are the ones building the most ambitious systems. The teams that handle them best are the ones that treat the relationship between agent reasoning and agent behavior as an engineering problem to be solved, not an assumption to be trusted.
